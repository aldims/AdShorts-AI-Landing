# Как не подмешать устаревшую озвучку в новый рендер: контракты и инвалидация в видеопайплайне

В редакторе коротких видео пользователь может заменить одну фразу, выбрать другой голос, переставить сцены, укоротить визуал или изменить оформление субтитров. На экране все выглядит правильно, но внутри системы продолжает существовать набор уже созданных файлов: озвучка отдельных сцен, общая голосовая дорожка, видео без субтитров, промежуточный микс и финальный рендер.

Самая неприятная ошибка в таком пайплайне выглядит безобидно: новый проект собирается из старого ассета, который формально существует и доступен по URL, но больше не соответствует текущему состоянию редактора.

Простое решение — удалять весь кеш после каждого изменения и всегда рендерить проект заново. Оно корректно, но быстро становится слишком дорогим. Более сложное решение — переиспользовать все, что еще подходит. Тогда задача перестает быть обычным кешированием: система должна уметь **доказать**, что конкретный ассет был создан из текущих входных данных.

Ниже — архитектура такого доказательства: авторитетный таймлайн, provenance аудиоассетов, fingerprint общего voice mix, явные системы координат таймингов и fail-closed выбор стратегии рендера.

## Почему существование файла ничего не доказывает

Представим проект из пяти сцен. У каждой есть текст, голос, визуал, длительность и субтитры. После первого рендера сохранены:

- аудиофайлы озвучки сцен;
- общая голосовая дорожка;
- чистые визуальные фрагменты;
- финальное видео;
- тайминги слов и субтитров.

Теперь пользователь меняет текст третьей сцены. Старый аудиофайл по-прежнему существует. Его можно скачать, декодировать и проиграть. Но он уже не является корректной озвучкой третьей сцены.

Проверка вида `if audio_url: reuse()` отвечает только на вопрос «есть ли файл». Нам нужны ответы на другие вопросы:

- для какого текста он был создан;
- каким голосом и на каком языке;
- относится ли он к отдельной сцене или ко всему проекту;
- какое окно общей дорожки было использовано;
- не изменился ли порядок сцен;
- совпадают ли границы таймлайна;
- не был ли ассет явно удален пользователем.

Поэтому в нашей реализации наличие ассета — необходимое, но недостаточное условие повторного использования.

## Сначала определяем владельца состояния

В реальном приложении состояние проекта редко живет в одном месте. Одновременно могут существовать:

1. текущий черновик в браузере;
2. сохраненные настройки проекта;
3. старый compatibility snapshot;
4. payload текущей операции редактирования;
5. контракт последнего успешного рендера.

Если каждый слой самостоятельно выбирает «самые полные» данные, результат становится недетерминированным. Особенно опасен fallback по принципу «текущий массив пустой, возьмем старый». Пустой массив может означать не отсутствие данных, а явное удаление всех сцен или ассетов.

Мы используем два правила:

**Текущая транзакция редактирования важнее сохраненного состояния.** Worker канонизирует timeline сразу на входе, и дальше все этапы получают одинаковый порядок сцен.

**Отсутствующее поле и явно пустое поле — разные состояния.** Если новый клиент вообще не передал поле, допустим compatibility fallback. Если он передал пустой массив, это авторитетная очистка, которую нельзя отменить данными из старого контракта.

Упрощенно выбор источника выглядит так:

```python
def select_timeline(current, saved):
    if "video_segments" in current:
        return current["video_segments"]  # в том числе []
    if "legacy_segments" in current:
        return current["legacy_segments"]
    if "video_segments" in saved:
        return saved["video_segments"]
    return saved.get("legacy_segments", [])
```

Главное здесь не название полей, а семантика присутствия. Явный empty state должен проходить через API, Worker и сохранение без попытки «починить» его устаревшим состоянием.

## Ассет без provenance — это просто файл

Для озвучки отдельной сцены мы храним не только ссылку на медиа. Рядом нужны данные, позволяющие проверить происхождение:

```text
SceneVoiceoverRef
    asset_id
    segment_index
    text_sha256
    voice_id
    language
    source_start
    source_end
    coordinate_space
```

Это не точная схема API, а минимальная модель контракта.

Озвучка сцены считается актуальной, только если:

- ассет существует;
- он не является общей голосовой дорожкой проекта;
- хеш текущего текста совпадает с сохраненным хешем;
- совпадает эффективный голос;
- совпадает язык;
- голос не отключен для этой сцены.

При изменении текста или голоса мы очищаем не только `asset_id`. Вместе с ним должны исчезнуть зависимые метаданные:

- длительность речи;
- начало и конец речи;
- тайминги слов;
- окно исходной дорожки;
- система координат;
- алиасы, оставшиеся от старых версий payload.

Атомарная инвалидация важна. Если удалить только URL, старые тайминги могут повлиять на длительность новой сцены. Если удалить только тайминги, старый файл может снова появиться после фонового refresh. Частично очищенное состояние часто опаснее явно отсутствующего.

## Почему хеша текста недостаточно

Для одиночной сцены набор `text + voice + language` уже дает полезный fingerprint. Для общей голосовой дорожки этого мало.

Допустим, тексты всех сцен остались прежними, но пользователь поменял местами вторую и третью сцены. Конкатенация тех же аудиофайлов в другом порядке даст другую дорожку. Или пользователь вручную изменил длительность сцены: текст не поменялся, но окно воспроизведения и расположение субтитров уже другие.

Поэтому fingerprint общего voice mix включает:

- id общего TTS-ассета;
- язык и глобальный голос;
- позицию сцены в текущем массиве;
- стабильный индекс сцены;
- хеш нормализованного текста;
- эффективный голос каждой сцены;
- режим длительности;
- `start`, `end` и `duration` сцены;
- окно аудиоисточника;
- систему координат этого окна;
- идентификатор голосового ассета сцены.

Канонический payload сериализуется со стабильным порядком ключей, после чего вычисляется SHA-256:

```python
canonical = {
    "version": "voice_mix.v1",
    "tts_asset_id": tts_asset_id,
    "language": language,
    "voice": global_voice,
    "segments": canonical_segments,
}

fingerprint = sha256(stable_json(canonical))
```

Хеш здесь не используется как средство защиты данных. Это компактная идентичность полного аудиоконтракта.

В fingerprint намеренно не входит идентификатор визуала. Замена фотографии не меняет произносимую дорожку. Она может заставить систему выбрать полный визуальный рендер, но не обязана инвалидировать корректный аудиомикс.

Это разделение позволяет не превращать любую правку в глобальную инвалидацию.

## Тайминги без системы координат похожи на корректные

Одна из самых коварных ошибок возникает после нарезки общей озвучки на ассеты сцен.

Пусть речь второй сцены находится в общей дорожке между `3.2` и `5.8` секунды. После вырезания получается отдельный файл длительностью `2.6` секунды. Слово с глобальными границами `3.4–3.8` внутри нового файла должно получить локальные границы `0.2–0.6`.

Если сохранить только числа без системы координат, невозможно понять, нужно ли вычитать `3.2` еще раз. Оба набора выглядят правдоподобно:

```text
global_audio: 3.4 -> 3.8
asset_local:  0.2 -> 0.6
```

Мы явно маркируем два пространства:

- `global_audio` — координаты внутри полной дорожки проекта;
- `asset_local` — координаты внутри отдельного ассета сцены.

Преобразование происходит один раз на границе, где из общей дорожки создается scene asset:

```python
def to_asset_local(word, source_start, asset_duration):
    start = clamp(word.start - source_start, 0, asset_duration)
    end = clamp(word.end - source_start, start, asset_duration)
    return Word(start=start, end=end, text=word.text)
```

После этого локальный ассет хранит локальные тайминги. Происхождение provider marks можно сохранить отдельно, но downstream-код больше не должен угадывать систему координат по величине чисел.

Та же идея полезна далеко за пределами видео: единицы измерения, timezone, coordinate space и ownership лучше делать частью типа, а не неявным соглашением между функциями.

## Кто управляет длительностью сцены

У сцены обычно есть несколько конкурирующих длительностей:

- длительность визуального файла;
- длительность озвучки;
- длительность речи без пауз;
- вручную выбранная длительность слота;
- длительность, сохраненная после последнего рендера.

Выбор максимального значения не всегда корректен. Пользователь мог осознанно оставить длинный визуальный хвост после окончания речи. Или наоборот — попросить синхронизировать видео строго с озвучкой.

Мы храним режим владения длительностью:

- `voiceover` — слот следует за воспроизводимой озвучкой;
- `visual` — сохраняется длительность визуала или явный пользовательский выбор.

Есть два инварианта:

1. фоновый refresh не перезаписывает длительность, которую пользователь выбрал явно;
2. слот не становится короче обязательного аудио, если оно должно быть воспроизведено полностью.

После разрешения длительности каждой сцены таймлайн строится заново одним проходом:

```python
cursor = 0

for scene in scenes:
    duration = resolve_duration(scene)
    scene.start = cursor
    scene.end = cursor + duration
    cursor = scene.end
```

Это сохраняет непрерывность после перестановки, удаления или изменения сцен. Границы речи при необходимости сдвигаются вместе со сценой, но не растягиваются автоматически до всей длины визуала.

## Быстрый рендер должен быть заслуженным состоянием

После появления точных контрактов можно безопасно выбирать частичный рендер. В нашей схеме есть четыре стратегии.

| Что изменилось | Стратегия | Обязательное условие |
| --- | --- | --- |
| Визуал сцены | Полный рендер | Безусловно |
| Только оформление субтитров | Наложение субтитров | Есть постоянный чистый визуал |
| Голос, язык, произносимый текст или звук сцены | Пересборка аудио поверх визуала | Чистым визуалом покрыты все сцены |
| Только музыка | Пересборка музыкального микса | Voice mix точно совпадает и есть чистый визуал |
| Неоднозначное состояние | Полный рендер | Безусловно |

Селектор можно представить так:

```python
if visual_changed:
    return FULL_RENDER

if subtitle_style_only and durable_clean_visual:
    return SUBTITLE_OVERLAY

if voice_or_text_or_language_changed:
    if clean_visual_coverage_complete:
        return AUDIO_REBUILD_REUSE_VISUAL
    return FULL_RENDER

if music_only and exact_voice_mix and durable_clean_visual:
    return MUSIC_REMIX

return FULL_RENDER
```

Здесь важен последний `return`. Неизвестное состояние не должно попадать в самый быстрый путь. Если система не смогла доказать безопасность оптимизации, она выбирает полный рендер.

## Render contract как снимок успешных входов

Fingerprint отвечает за идентичность аудио, но для воспроизводимого рендера нужен более широкий контракт. После успешной операции мы сохраняем структуру, связывающую сцены, тайминг, визуальные источники, настройки бренда и результат.

Строгая версия контракта проверяет:

- количество сцен;
- уникальность source index;
- конечные положительные интервалы;
- начало таймлайна с нуля;
- отсутствие разрывов и пересечений;
- соответствие медиавходов сценам;
- отсутствие неявно добавленных настроек;
- целостность визуального действия для каждой сцены.

Контракт не является «еще одной копией проекта». Это доказательство того, из каких входов был получен конкретный успешный результат.

Поэтому восстановление ассетов из него допустимо только при полном соответствии текущему проекту. Если совпали четыре сцены из пяти, нельзя тихо восстановить четыре старых аудиофайла и сгенерировать пятый: такой гибрид требует отдельного явно поддерживаемого сценария. В текущей модели неоднозначное частичное восстановление отклоняется.

## Почему явная очистка важнее автоматического восстановления

Recovery-механизмы часто пытаются быть слишком полезными. После сбоя или старого клиента система видит отсутствующую озвучку и находит подходящий ассет в контракте последнего рендера.

Это полезно, пока отсутствие действительно означает потерю данных. Но если пользователь отключил голос или отредактировал сцену, автоматическое восстановление возвращает то, что он уже удалил.

Поэтому наличие поля `segment_voiceover_assets` является сигналом авторитетного текущего состояния. Даже если значение пустое, legacy-контракт не может вернуть ассеты. Recovery запускается только для совместимых старых состояний и только при совпадении идентичности сцены, текста, голоса и языка.

Общий принцип:

> Явное новое состояние сильнее неявного старого fallback.

## Какие regression-тесты действительно защищают пайплайн

Happy path с неизмененным проектом мало что говорит о корректности инвалидации. Основную ценность дают тесты на границах состояния.

Мы проверяем, что:

- изменение текста запрещает использовать старый project TTS;
- смена языка инвалидирует аудио, даже если идентификатор голоса выглядит одинаково;
- перестановка сцен меняет fingerprint общего voice mix;
- изменение идентификатора scene voiceover меняет fingerprint;
- изменение только визуального asset id не меняет аудиофингерпринт;
- глобальные окна речи становятся локальными ровно один раз;
- пустой текущий timeline не оживляет старые сцены;
- явно очищенный список voiceover assets не восстанавливается из контракта;
- выбранная пользователем длительность переживает server refresh;
- отсутствие полного clean visual coverage запрещает частичную пересборку аудио;
- разрыв, пересечение или невалидная длительность делают render contract невалидным.

Полезно разделять тесты по уровням:

1. чистые функции проверяют fingerprint и strategy selection;
2. API-тесты проверяют выбор авторитетного состояния и recovery;
3. Worker-тесты проверяют генерацию ассетов и преобразование таймингов;
4. frontend-тесты проверяют инвалидацию черновика и сохранение пользовательского ownership.

Так ошибка локализуется ближе к месту, где нарушился контракт, а не проявляется только несовпадением финального видео.

## Что в итоге оказалось важнее самого кеша

Переиспользование сгенерированного медиа сначала является задачей корректности и только потом оптимизацией производительности.

Для предсказуемого пайплайна нам понадобились семь свойств:

1. один авторитетный таймлайн на операцию;
2. различие между отсутствующим и явно пустым состоянием;
3. provenance рядом с каждым сгенерированным ассетом;
4. fingerprint всех входов, влияющих на повторно используемый результат;
5. явная система координат таймингов;
6. явный владелец длительности;
7. fail-closed выбор стратегии рендера.

После этого частичный рендер перестает быть набором оптимистичных `if`. Каждая быстрая ветка получает проверяемые предусловия, а полный рендер остается корректным состоянием по умолчанию.

Эти принципы применимы не только к генерации видео. Любой пайплайн с дорогими производными ассетами — аудио, превью, транскодинг, документы или ML-результаты — выигрывает, когда кеш хранит не только результат, но и доказательство его происхождения.
