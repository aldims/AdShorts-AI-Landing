# Из текста в вертикальное видео: архитектура пайплайна со сценами, озвучкой и субтитрами

На входе — несколько предложений или одна идея. На выходе — готовый ролик 9:16: с видеорядом, озвучкой, музыкой и субтитрами. Между этими точками находится не одна нейросеть, а небольшой медиапродакшен, разложенный на сервисы и данные.

Под «готовым роликом» здесь понимается [вертикальное видео с собранными сценами, озвучкой и субтитрами](https://adshortsai.com/examples/), а не отдельный сырой фрагмент из text-to-video модели.

Главная сложность такого продукта не в том, чтобы один раз получить красивый MP4. Сложность — повторяемо собирать видео после правок: заменить фразу, переставить сцены, выбрать другой голос, изменить длительность кадра или оформление субтитров и не получить рассинхрон.

В статье разберу минимальную архитектуру такого пайплайна: как представить сцены, кто управляет временем, где хранить происхождение ассетов, как синхронизировать речь и субтитры и когда разрешать частичный рендер.

## Что должна гарантировать система

Предположим, пользователь ввел тему: «Почему короткие видео не набирают просмотры». Система предложила четыре сцены:

1. короткий вопрос в первом кадре;
2. объяснение одной причины;
3. конкретную проверку;
4. итоговое действие.

Для каждой сцены нужны текст, визуал, голос и место на общей шкале времени. Уже здесь появляются пять инвариантов:

- сцены идут в том порядке, который видит пользователь;
- озвучка соответствует текущему тексту и голосу;
- субтитры попадают в произносимые фразы;
- ни одна сцена не начинается раньше окончания предыдущей;
- повторный рендер использует только те ассеты, чье происхождение можно доказать.

Если хотя бы один инвариант нарушен, отдельные компоненты могут выглядеть исправными, а итоговый ролик — нет. Например, новый текст окажется рядом со старой озвучкой или субтитры второй сцены будут размечены во времени всего проекта, хотя аудиофайл уже вырезан отдельно.

## Пайплайн целиком

Упрощенно процесс выглядит так:

```text
идея или сценарий
        |
        v
план сцен и текст каждой сцены
        |
        +----> подбор или генерация визуалов
        |
        +----> TTS и тайминги речи
        |
        v
авторитетный таймлайн проекта
        |
        +----> субтитры
        +----> музыка и звуковой микс
        +----> композиция 9:16
        |
        v
проверка контракта -> рендер -> MP4
```

Важная деталь: это не линейная очередь, в которой каждый этап безусловно запускает следующий. Визуалы и озвучка могут создаваться параллельно. После пользовательской правки нужно пересчитать только затронутые производные данные, но результат снова должен пройти общую проверку.

## Сцена как основная единица данных

Склеивать пайплайн вокруг массива URL неудобно. URL ничего не говорит о том, почему файл появился и можно ли использовать его после изменения проекта. Нужен контракт сцены.

Минимальная модель может выглядеть так:

```typescript
type Scene = {
  id: string;
  position: number;

  text: string;
  language: string;
  voiceId: string | null;

  visualAssetId: string | null;
  voiceAssetId: string | null;

  durationOwner: "voiceover" | "visual";
  start: number;
  end: number;

  speechWindow: {
    start: number;
    end: number;
    coordinateSpace: "global_audio" | "asset_local";
  } | null;
};
```

Здесь пользовательский текст и выбранный голос — входные данные. Аудиофайл, тайминги речи и финальные границы сцены — производные данные. Это различие определяет правила обновления.

Если пользователь изменил текст, система не должна сохранять прежний `voiceAssetId` только потому, что файл по этому адресу еще существует. Если изменен только визуал, озвучка может остаться актуальной. Если сцены переставлены, сами аудиофайлы могут быть корректны, но общий звуковой микс и таймлайн уже изменились.

## Шаг 1. Из сценария получить план сцен

Первая стадия должна выдавать не литературный текст, а структуру, пригодную для последующей сборки.

Для короткого ролика полезно ограничить контракт генератора:

```json
{
  "title": "Почему Shorts не набирают просмотры",
  "scenes": [
    {
      "purpose": "hook",
      "voiceover": "Ролик получает показы, но зрители уходят в первую секунду?",
      "visual_query": "analytics retention chart"
    },
    {
      "purpose": "explanation",
      "voiceover": "Чаще всего причина в первом кадре: он не обещает понятный результат.",
      "visual_query": "vertical video first frame"
    }
  ]
}
```

Практические ограничения важнее сложного промпта:

- фиксированный максимум сцен;
- одна мысль на сцену;
- отдельный текст для речи, а не HTML или Markdown;
- явное назначение сцены;
- ограничение длины реплики;
- стабильный идентификатор после сохранения.

Модель может вернуть идею с разным количеством слов. Поэтому длительность нельзя надежно назначить до получения озвучки или хотя бы оценки речи.

## Шаг 2. Создать визуалы независимо от времени

Визуальный слой может быть фотографией, видеофрагментом, сгенерированным изображением или пользовательским файлом. На этой стадии достаточно связать постоянный `visualAssetId` со сценой и сохранить метаданные источника.

Не стоит сразу записывать длительность исходного файла как длительность сцены. Десятисекундный клип можно использовать в трехсекундном слоте, а изображение вообще не имеет собственной временной длины.

Поэтому полезно разделить:

- свойства источника: разрешение, исходную длительность, формат;
- действие в сцене: crop, scale, pan, trim, loop;
- длительность слота на итоговом таймлайне.

Для вертикального видео дополнительно нужен детерминированный способ привести вход к 9:16. Например:

```text
если источник уже близок к 9:16:
    crop по безопасной центральной области
иначе если важен весь кадр:
    fit внутри 9:16 + фон
иначе:
    cover 9:16 с сохранением выбранной точки фокуса
```

Выбор режима должен храниться в проекте. Иначе preview и серверный рендер могут принять разные решения для одного файла.

## Шаг 3. Генерировать озвучку вместе с происхождением

У TTS есть два распространенных варианта организации.

| Подход | Плюсы | Минусы |
| --- | --- | --- |
| Одна дорожка на весь проект | Естественные общие паузы, меньше запросов | Сложнее менять одну сцену и локализовать тайминги |
| Отдельная озвучка каждой сцены | Проще редактировать, кешировать и переставлять сцены | Нужно управлять паузами и собирать общий микс |

Оба варианта рабочие. Важно, чтобы рядом с результатом хранились входы, от которых он зависит:

```typescript
type VoiceAssetProvenance = {
  assetId: string;
  sceneId: string;
  textHash: string;
  voiceId: string;
  language: string;
  provider: string;
  duration: number;
};
```

Проверка актуальности становится явной:

```typescript
function canReuseVoice(scene: Scene, asset: VoiceAssetProvenance) {
  return asset.sceneId === scene.id
    && asset.textHash === hash(normalize(scene.text))
    && asset.voiceId === scene.voiceId
    && asset.language === scene.language;
}
```

Это не попытка превратить медиапайплайн в блокчейн. Метаданные нужны для простого ответа на вопрос: создан ли этот файл из текущих входов?

## Шаг 4. Назначить владельца длительности

У сцены может быть несколько правдоподобных длительностей:

- длина исходного видео;
- длина TTS-файла;
- длина речи без пауз;
- значение, выбранное пользователем;
- значение после предыдущего рендера.

Функция `max()` здесь не решает продуктовую задачу. Пользователь может захотеть оставить визуальный хвост после речи или, наоборот, завершить сцену сразу вместе с голосом.

Мы используем два режима владения:

- `voiceover` — длительность слота следует за воспроизводимой озвучкой;
- `visual` — сохраняется выбранная длительность визуала, если обязательная речь помещается внутри.

Упрощенное разрешение длительности:

```typescript
function resolveDuration(scene: Scene, voiceDuration: number | null) {
  if (scene.durationOwner === "voiceover" && voiceDuration != null) {
    return voiceDuration;
  }

  const visualDuration = scene.end - scene.start;
  return Math.max(visualDuration, voiceDuration ?? 0);
}
```

После этого границы всех сцен лучше перестраивать одним проходом, а не исправлять точечно:

```typescript
let cursor = 0;

for (const scene of orderedScenes) {
  const duration = resolveDuration(scene, getVoiceDuration(scene));
  scene.start = cursor;
  scene.end = cursor + duration;
  cursor = scene.end;
}
```

Так перестановка, удаление или изменение одной сцены не оставляет скрытых разрывов и пересечений дальше по таймлайну.

## Шаг 5. Не путать глобальные и локальные тайминги

Представим, что речь второй сцены находится в общей TTS-дорожке между `3.2` и `5.8` секунды. После нарезки появляется отдельный аудиофайл длиной `2.6` секунды.

Слово с глобальными границами `3.4–3.8` внутри нового файла должно получить локальные границы `0.2–0.6`.

```text
global_audio: 3.4 -> 3.8
asset_local:  0.2 -> 0.6
```

Если сохранить только два числа, downstream-код не узнает, нужно ли вычитать начало сцены. Поэтому система координат должна быть частью контракта, а преобразование — выполняться ровно один раз:

```python
def to_asset_local(word, source_start, asset_duration):
    start = clamp(word.start - source_start, 0, asset_duration)
    end = clamp(word.end - source_start, start, asset_duration)
    return Word(start=start, end=end, text=word.text)
```

После нарезки scene asset хранит только локальные тайминги. Когда сцена ставится на общий таймлайн, к ним добавляется `scene.start`.

Этот контракт одновременно решает две задачи:

- субтитры появляются в правильной сцене;
- preview и render не применяют одно смещение дважды.

## Шаг 6. Построить субтитры из тех же временных данных

Субтитры не должны иметь отдельную независимую шкалу времени. Источник истины тот же, что используется для аудио.

Если TTS-провайдер возвращает тайминги слов, их можно сгруппировать в короткие фразы с ограничениями:

- не более двух строк;
- ограничение количества символов;
- перенос по смысловой границе;
- минимальное время показа;
- запрет выхода за границы сцены.

Если точных таймингов слов нет, безопасный fallback — распределить текст внутри окна речи сцены, а не по полной длине ролика.

```typescript
type SubtitleCue = {
  sceneId: string;
  start: number; // глобальное время проекта
  end: number;
  text: string;
};
```

Оформление субтитров — отдельный слой: шрифт, размер, позиция, обводка, фон и анимация не должны менять текстовые тайминги. Это позволяет повторно наложить стиль без пересборки визуалов и TTS.

## Шаг 7. Рендерить слоями, а не одним непрозрачным шагом

Финальную композицию удобно мыслить как набор постоянных слоев:

1. чистый визуальный таймлайн;
2. голосовая дорожка;
3. музыка и дополнительные звуки;
4. субтитры и графика;
5. кодирование итогового MP4.

Логическая схема рендера:

```python
visual = compose_vertical_scenes(scenes, resolution=(1080, 1920))
voice = compose_voice_track(scenes, timeline)
audio = mix_audio(voice, music, scene_sounds)
subtitles = build_subtitle_layer(cues, style)

result = encode(
    video=overlay(visual, subtitles),
    audio=audio,
    fps=30,
)
```

Реализация может использовать FFmpeg, Remotion или собственный renderer. Архитектурный принцип не меняется: каждый слой получает проверенный контракт, а не собирает проект заново из разрозненных полей.

Перед рендером полезно проверять:

- таймлайн начинается с нуля;
- интервалы конечные и положительные;
- между соседними сценами нет разрывов и пересечений;
- у каждой сцены есть визуальный источник;
- актуальная озвучка соответствует тексту, голосу и языку;
- субтитры не выходят за общую длительность;
- входы рендера относятся к одной версии проекта.

## Что делать после пользовательских правок

Полный рендер после каждого изменения надежен, но дорог. Частичный рендер быстрее, если система может доказать его безопасность.

| Изменение | Что пересобрать |
| --- | --- |
| Текст, голос или язык сцены | Озвучку, тайминги, субтитры и аудиомикс |
| Визуал сцены | Визуальный слой и финальную композицию |
| Только стиль субтитров | Слой субтитров поверх чистого визуала |
| Только музыку | Аудиомикс, если голосовая дорожка не изменилась |
| Порядок или длительность сцен | Таймлайн, микс и финальную композицию |

Селектор стратегии лучше строить по принципу fail closed:

```python
if visual_changed:
    return FULL_RENDER

if subtitle_style_only and clean_visual_exists:
    return SUBTITLE_OVERLAY

if voice_inputs_changed and clean_visual_covers_all_scenes:
    return AUDIO_REBUILD_REUSE_VISUAL

if music_only and exact_voice_mix_matches:
    return MUSIC_REMIX

return FULL_RENDER
```

Самый быстрый путь не является состоянием по умолчанию. Его нужно заслужить выполненными предусловиями.

## Как переживать повторы и сбои

Генерация медиа связана с сетью, внешними провайдерами и длительными задачами. Worker может получить повторное сообщение после таймаута, а пользователь — нажать кнопку еще раз.

Поэтому операции должны быть идемпотентными относительно версии входов:

```text
operation key = project_id
              + scene_id
              + operation_type
              + input_fingerprint
```

Повтор с тем же fingerprint может вернуть уже готовый результат. Повтор с новым fingerprint обязан создать новую производную версию, а не перезаписать происхождение старого файла.

Практически полезно разделять статусы:

- `queued` — задача поставлена;
- `running` — Worker начал операцию;
- `succeeded` — результат и provenance сохранены атомарно;
- `failed_retryable` — можно повторить;
- `failed_terminal` — входы некорректны или провайдер окончательно отклонил запрос;
- `superseded` — результат завершился, но проект уже изменился.

Последний статус особенно важен. Успешно созданный файл не всегда является актуальным результатом для текущего проекта.

## Наблюдаемость: логировать не только ошибку рендера

Сообщение «FFmpeg завершился с кодом 1» редко объясняет причину пользователю или разработчику. Для каждой операции полезно записывать:

- `project_id` и версию проекта;
- `scene_id`;
- тип операции;
- fingerprint входов;
- выбранную стратегию рендера;
- идентификаторы использованных ассетов;
- длительности сцен, голоса и результата;
- причину отказа от fast path;
- время каждого этапа.

Тогда можно отличить сбой кодирования от более ранней проблемы: отсутствующего визуала, устаревшей озвучки, несовместимых таймингов или неполного clean visual слоя.

## Тесты, которые ловят реальные рассинхроны

Обычный тест «создать проект и получить MP4» проверяет happy path, но почти не защищает редактор. Полезнее проверять изменения состояния.

Минимальный набор regression-тестов:

1. изменение текста запрещает использовать старую озвучку;
2. смена языка инвалидирует голосовой ассет даже при том же `voiceId`;
3. перестановка сцен перестраивает общий таймлайн и микс;
4. изменение только визуала не инвалидирует корректный TTS сцены;
5. глобальные тайминги становятся локальными ровно один раз;
6. пользовательская длительность не исчезает после обновления данных с сервера;
7. пустой текущий список сцен не восстанавливает старый compatibility snapshot;
8. отсутствие чистого визуального слоя запрещает частичный аудиорендер;
9. разрыв или пересечение сцен блокирует финальный рендер;
10. результат старой фоновой задачи получает статус `superseded` и не заменяет новую версию.

Тесты удобно разделить по слоям:

- чистые функции: длительность, fingerprints и выбор стратегии;
- API: авторитетное состояние, версии и recovery;
- Worker: создание ассетов и преобразование таймингов;
- frontend: инвалидация производных данных после редактирования;
- интеграционный тест: сборка короткого проекта с последующей правкой одной сцены.

## Итоговая схема владения данными

В устойчивом пайплайне каждый класс данных имеет одного владельца:

| Данные | Владелец |
| --- | --- |
| Текст, порядок и настройки сцен | Текущая версия проекта |
| Границы сцен | Авторитетный таймлайн |
| Происхождение TTS | Voice asset metadata |
| Локальные тайминги слов | Аудиоассет сцены |
| Глобальные субтитры | Сборщик таймлайна |
| Решение о частичном рендере | Strategy selector |
| Входы успешного MP4 | Render contract |

Эта дисциплина кажется избыточной, пока продукт только генерирует первый ролик. Она становится необходимой, как только появляется редактор и пользователь ожидает, что небольшая правка изменит именно выбранную часть проекта.

Если свести подход к короткому чек-листу:

1. хранить проект как сцены, а не как набор URL;
2. отделять входные данные от производных ассетов;
3. держать один авторитетный таймлайн;
4. сохранять provenance рядом с каждым сгенерированным файлом;
5. маркировать систему координат таймингов;
6. явно назначать владельца длительности;
7. разрешать fast path только после проверки предусловий;
8. отклонять результаты фоновых задач для устаревшей версии проекта;
